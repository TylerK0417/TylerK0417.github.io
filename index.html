QUANTDASH Trading Bot

Machine Learning QUANTDASH Trading Bot:

Built from my interest in equity trading and building an auto trading bot that stored trades for machine learning

• Developed a Python bot streaming tick-level equities/crypto (Alpaca API), storing Parquet data for SQL-based analysis

• Engineered SQL based ETL pipelines and scikit-learn/TensorFlow models with walk-forward validation and risk managed baselines

• Backtested 5M + rows across 50+ equities over 5 years to optimize strategies and features; executed live paper trades

How it works (demo)

  1. We launch the bot; it initializes the Alpaca paper client, loads any existing positions from the paper account into the local portfolio, and computes initial stop/trailing levels for each open position.
  2. The optional Live Dashboard spins up (when --dashboard is used), and the bot prints the active config: symbols universe, bar interval, risk and trail percentages, and timeouts for limit and cancel.
  3. Before trading, the bot reconciles open orders from Alpaca and populates its “open-order guard” so it won’t double-send if something is already working at the broker.
  4. The main loop starts (every N seconds). For each symbol, it ingests the latest OHLCV bars, computes the MA(1/2) crossover signal, and derives an ATR value used both for initial stop distance and position sizing.
  5. If a long entry signal appears and we’re flat, the bot sizes the position using risk controls (ATR-based stop distance, buying power/notional caps) and submits a LIMIT order first. For equities, it uses DAY TIF; for crypto, GTC.
  6. While a LIMIT is working, the bot’s duplicate-order guard (keyed by symbol+side and refreshed from Alpaca) prevents any re-entrant tick or stop logic from posting a second order. Every submit also includes a unique client_order_id for broker-side idempotency.
  7. If the LIMIT hasn’t filled within the configured limit timeout, the bot issues a cancel, polls for a terminal state, and then sends a fallback order for the exact remaining quantity: - Equities: a “marketable LIMIT” with extended_hours=True (acts like market after-hours). - Crypto: a true MARKET order (24/7).
  8. For exits (either a flat signal or a stop/take-profit hit), the bot mirrors the same flow: submit LIMIT, wait, cancel, then fallback for the remainder, again protected by the same dedupe guard so we don’t double-sell or accidentally short.
  9. On every loop, the bot ratchets trailing stops: when a bar makes a new high against the trail base, it moves the stop up (never down) by the configured trail percent (or ATR mode, per your settings). Stop/TP breaches trigger immediate exits.
  10. All actions are logged: each place/cancel/fill path prints to the console and is also written to a persistent CSV (timestamp, symbol, side, qty, order IDs, and prices), so you can show auditability in the video.
  11. The bot enforces the “one position per symbol” rule: if already long and the signal stays long, it does nothing; if flat and the signal stays flat, it does nothing—only state changes trigger orders.
  12. When you stop the process, the portfolio state (positions and their current stops/trail bases) is saved, so a restart will pick up where it left off without duplicate orders or lost risk settings.

Next Steps: For Machine Learning and Live Trading

  • 1. ML-driven signals: move from rules to a probability “signal scorer” (tree-based first; later lightweight transformers) that predicts profitable trades.
  • 2. Richer features: add regime context (breadth, volatility, macro calendar) and selective alt-data (sentiment, options skew) to boost robustness.
  • 3. Adaptive risk & sizing: scale position size with model confidence and volatility; enforce exposure/correlation caps and hard kill-switches
  • 4. Rolling retrain & promotion: nightly/weekly walk-forward retrains; promote only if out-of-sample Sharpe/hit-rate improve; auto-rollback and drift monitoring.
  • 5. Execution & ops hardening: smarter marketable-limit logic (RTH/AH aware), partial-fill handling, live dashboards, and CI/CD that blocks deploys on metric regressions.
